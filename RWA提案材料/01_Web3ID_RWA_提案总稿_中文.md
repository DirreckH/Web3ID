# Web3ID for RWA 提案总稿

## 1. 项目概览

`Web3ID` 是一套面向真实系统运行的 Web3 身份与合规执行基线。我们希望把 `programmable identity`、`proof-aware access`、`auditability` 与 `governed risk` 放进同一条可解释、可验证、可治理的系统主链路中。

面向 `RWA` 领域，Web3ID 的核心目标不是再造一个静态白名单系统，而是把“谁可以在什么条件下持有、转移、购买某类现实世界资产”变成一套可持续运行的状态与规则系统。项目当前已经具备公开可访问的原型站点、可浏览的前端 demo、完整的系统文档和 RWA 购买流程演示。

## 2. RWA 领域的核心问题

当前 RWA 赛道存在一个长期未被真正解决的矛盾: 现实世界资产需要合规约束，但链上代币天然倾向自由流转。

这个矛盾至少表现为四个层面:

1. 资产可以链上转移，但现实世界的投资资格、地区限制、制裁约束和合格投资人规则并不会因此消失。
2. 大多数 RWA 系统只在入口做一次 KYC 或白名单校验，之后不持续追踪资格变化、风险变化和监管状态变化。
3. 不同司法辖区的资产规则不一致，但链本身是全球开放的，缺少一层真正可编程的身份规则层。
4. 传统白名单地址方案过度依赖中心化托管、地址冻结和手工控制，牺牲了链上可组合性与流动性。

## 3. Web3ID 的解决方案

Web3ID 针对 RWA 采用的不是“白名单地址”思路，而是“白名单状态”思路。

传统做法是:

```text
if address in whitelist:
  allow transfer
```

Web3ID 的做法是:

```text
if verify(Web3ID proof, rule_set):
  allow transfer
```

这意味着:

- 合规约束绑定的是状态与证明，而不是某一个永久地址
- 用户可以在不暴露底层身份细节的前提下证明自己满足规则
- 规则可以动态更新，验证可以持续发生
- RWA 资产的链上流动从“人工托管放行”转向“规则驱动放行”

## 4. 核心技术机制

### 4.1 Proof-aware transfer gating

Web3ID 让每一笔高敏感动作都建立在 proof-aware access 上。对于 RWA 购买、持仓调整、跨场景转移等操作，系统要求动作在进入执行前先通过规则、证明与上下文验证，而不是单靠某个地址曾经通过审核。

### 4.2 动态合规

RWA 的合规不是一次性认证，而是持续性验证。Web3ID 支持在动作发生时重新检查:

- 当前合规状态
- 当前风险等级
- 当前司法辖区规则
- 当前 credential / proof 是否仍然有效

这使系统能应对资格过期、风险上升、地址污染、规则更新等现实问题。

### 4.3 子身份隔离

Web3ID 中的 `RootIdentity`、`SubIdentity` 与 `SubjectAggregate` 让系统可以把控制权、场景隔离和主体归并明确分层。

对 RWA 而言，这带来两个直接收益:

- 不同资产、策略、辖区、风险等级可以分配到不同子身份路径
- 资产污染、风险传导和混用问题不必依赖“一地址一世界”的笨重方案

### 4.4 跨司法辖区规则引擎

不同国家和地区的 RWA 规则可以被抽象成不同 `rule_set`。用户不必手动理解复杂法规，而是通过本地钱包或系统侧生成证明，在动作发生时完成规则匹配。

如果不满足规则，系统会拒绝动作并保留限制原因与审计证据。

### 4.5 审计与治理闭环

Web3ID 不是单纯的前端购买演示。系统把风险信号、状态判断、策略执行、结果保留和审计导出放在同一条主链路里，并明确限制 AI 只能停留在 `advisory-only` 边界内，不能越权写正式状态。

## 6. Why Now: 为什么这个方案适合当前 RWA 阶段

RWA 赛道真正缺的不是再多一个链上资产商城，而是一个能把身份、合规、持续性验证和可流动性结合起来的底层执行层。

Web3ID 的定位正好补这个缺口:

- 它不把合规简单理解为人工审核和静态白名单
- 它不把身份等同于单个地址
- 它不把风控做成黑箱打分
- 它把状态、策略、审计与解释整合进一个正式系统模型

对于 RWA 平台、发行方和合规合作方而言，这种能力比“再做一个前端交易壳子”更接近底层基础设施。

## 7. Non-token 经济与激励设计

本项目不以发币叙事为前提，`tokenomics` 章节采用 `non-token economic design` 方式表达。

### 7.1 收费模型

项目未来可以形成三层收入结构:

1. 资产发行与准入服务费
   - 面向 RWA 发行方提供身份规则配置、准入策略配置、发行前验证与持续性合规服务
2. 交易与规则执行服务费
   - 对高敏感动作的 proof verification、规则匹配、风控评估和审计封装收取服务费
3. 企业级合规与审计 SaaS
   - 面向机构客户提供 dashboard、审计导出、规则编排、操作员工作流与治理控制平面

### 7.2 价值流转

价值并不围绕代币价格形成，而是围绕真实服务价值形成:

- 发行方为降低合规运营成本付费
- 平台方为获得动态准入能力和风险可解释性付费
- 机构投资者为更低摩擦、可验证、跨辖区的准入体验受益
- 审计、合规与治理参与方通过接入统一状态与证据链降低协作成本

### 7.3 激励设计

激励设计重点放在生态协作，而非代币激励:

- 发行方获得更可持续的合规流动性
- 平台获得更低的人工作业成本和更好的可监管性
- 用户获得更少暴露、更多可迁移性的准入证明能力
- 审计与监管协作方获得更清晰的证据链与回放能力

## 8. BNB Chain 适配说明

本项目当前公开 demo 主要以产品原型和系统能力展示为主，但技术路线可以自然对接 `BNB Chain` 场景。

原因在于:

- 当前系统已经支持 EVM controller family
- 合规模型和 proof-aware access 是链上动作前置逻辑，不依赖单一业务页面
- RWA 购买和准入逻辑可以映射到 BNB Chain 上的资产发行、持仓和交易约束场景

提交比赛时，建议把该部分表述为:

> Web3ID is building the identity, rule verification, and compliance execution layer that can be applied to BNB Chain based RWA issuance and circulation scenarios.

不要表述成“已经完成 BNB Chain 生产部署”，除非届时已有真实结果。
